[Action Required: Windows XP OS users] End of Support for 3DES Cipher


For security reasons, we have discontinued the support for 3DES Cipher. After this discontinuation of support for 3DES cipher, machines running on Windows XP, that do not support any more recent and secure ciphers than 3DES, will not be able to access the Sign In and My Account pages.

If you're a Windows XP user, we strongly recommend upgrading your OS.

Accessing Storage with Relative Path

webOS Signage supports three types of relative paths for accessing the local storage, as listed below:

  • file://internal

  • ./content

  • http://127.0.0.1:9080

Linux absolute file path cannot be used by SCAP API on webOS Signage.

For each type of relative path, there is a set of allowed users and specific purposes as described below.

file://internal

  • User: The SCAP API

  • Purpose: For passing a file path to a SCAP API method.

  • Note: This path type is used only as a parameter to a SCAP API.

./content

  • User: HTML5 code, JavaScript code (NOT the SCAP API)

  • Purpose: For accessing files or directories that are created by the SCAP library.

  • Note: The absolute path of ./content depends on your app location, i.e. whether your app is in the local storage or is on a USB flash drive.

    • Local: Private path.

    • USB: [USB_root]:procentric\scap\application\content

    • Remote: Not applicable. The absolute path would become [remote_path]/content, which does not exist.

http://127.0.0.1:9080

  • User: Apps on a USB flash drive or on a remote server.

  • Purpose: For accessing files or directories in the signage's local storage.

  • Note: This path is identical to the ./content directory used by the apps in the local storage.

Note that the SCAP library cannot access the web app's storage area, as illustrated below. Thus the SCAP library cannot access your app's index.html file. The SCAP APIs can only access the paths it can parse; file://internal, http:// and https://.

EXAMPLE

Here are some code examples to help you get some idea of using relative paths with the SCAP library. We have provided some examples that are invalid due to the incorrect usage of the relative path. Click the invalid label to see why the particular code example is invalid.

Valid

var successCb = function(cbObject){console.log("SUCCESS") };
var failureCb = function(cbObject) { console.error("ERROR") };
var listOption = { path: "file://internal/sampleDirectory" };
new Storage.listFiles(successCb, failureCb, listOption);

Valid

document.getElementById("image").src = "./content/sampleDir/image.jpg"

Invalid

var successCb = function(cbObject) { console.log("SUCCESS") };
var failureCb = function(cbObject) { console.error("ERROR") };
var listOption = { path: "./content/sampleDirectory" };
new Storage.listFiles(successCb, failureCb, listOption);

The SCAP library cannot parse the given parameter, which starts with ./content. The SCAP library only accepts the path starting with file://, http:// and https://.

Invalid

document.getElementById("image").src = "file://internal/sampleDir/image.jpg"

Standard HTML cannot understand the file://internal path. This path type can only be used with the SCAP API. For HTML to use file, ./content shall be used.

Invalid

var successCb = function(cbObject) { console.log('SUCCESS') };
var failureCb = function(cbObject) { console.error('ERROR') };
var listOption = { path: "/some/of/linux/path/sampleDirectory" };
new Storage.listFiles(successCb, failureCb, listOption);

The SCAP library cannot parse the given parameter, which is a Linux file path. Linux file path is unacceptable. The SCAP library only accepts the path starting with file://, http:// and https://.

Invalid

document.getElementById("image").src = "/some/of/path/sampleDir/image.jpg"

The HTML code in your app can access only the directories in your app folder AND the ./content directory in webOS Signage.
